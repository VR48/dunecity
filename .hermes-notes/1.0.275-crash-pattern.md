# Crash pattern across versions 1.0.269 — 1.0.275

Tornie sent two more crash dumps from v1.0.275 (mod Tornie and mod
vanilla). Same crash offset, same crash trigger pattern. Advanced
Windtrap suspicion is **falsified** by these logs.

**Crash address** in both v1.0.275 logs: `0x00007FF7086054F8` and
`0x00007FF7__054F8` — both have low 16 bits = `0x54F8`, the same offset
in dunecity.exe as every previous log (1.0.269 → 1.0.275). The exe base
address varies with ASLR; the offset within the file is constant.

**Tornie's hypothesis was wrong**: "out of bound Advanced Windtrap" is
the cause Tornie remembered from 1.0.255-258 history, not the current
crash. The 1.0.275 logs show:

```
GFXManager: Loaded Tornie_AdvancedWindtrap_gfx.png as CornerFlag
   (single static 48x48 PNG, cross-house, DuneCity 1.0.254)
```

The loader succeeds for both mods. The crash happens AFTER the gfx
load, during `Now playing DUNE4.ADL!` / `DUNE5.ADL!` (mission 2
Harkonnen transition).

**Deviator is also fine**:
```
GFXManager: Loaded Deviator.png (palette-indexed)
```
loader succeeds in both logs.

**HeraldRebels banner is fine**:
```
GFXManager: HeraldRebelsMask.png loaded — companion banner mask for
HOUSE_REBELS
```
loader succeeds in both logs.

**Green-grid Rebels placement** has NOT been reproduced in these two
logs (Tornie played Atreides in both). My earlier hypothesis
("add `isWithinBuildRange` check in `House::placeStructure`") remains
un-validated.

## Real cause still un-identified

Bytecode reverse-engineering done earlier pointed at offset `0x140045480`
which compiles to a nested loop with a `cmp %dil, 0x1a(%rdx)` OOB read.

Earlier 1.0.274 log explicitly showed:
```
Game::init: scenario load failed (gameType=3, mission=22):
D:\a\dunecity\dunecity\src\misc\draw_util.cpp:608:
mapSurfaceColorRange(): Cannot copy image
(SDL_ConvertSurface failed: Empty destination palette)!
```

That throw has a `std::runtime_error` type which becomes `0xE06D7363`
in Windows exception encoding. BUT the v1.0.275 crash is `0xC0000005`
which is a true access violation (not a C++ throw). So `mapSurfaceColorRange`
may be one cause path but not the only one.

It is possible the v1.0.275 crash is a *different* exception than the
v1.0.274 one (different offset within the throw chain, different
cause). Without a Windows debugger attached to the live exe I cannot
map the 0xC0000005 to a specific instruction.

## What Tornie needs to do

To progress, one of:

1. **Run the game under WinDbg** so the exception site is resolved to
   a function name + line number (`DUNE-CITY!Class::method at line NNN`).
2. Or: install `gflags` + a debugger that writes crash context on
   exception, so the .dmp contains structured exception info.
3. Or: send a build with `SDL_Log` instrumentation at the suspected
   crash site (draw_util.cpp:608 + AdvancedWindTrap::init +
   Map::okayToPlaceStructure) so the next crash tells us which
   function was called before the exception.

## DO NOT DO

- Do NOT push another `blitToScreen null guard` (already shipped
  1.0.270, 1.0.272, 1.0.273 — none of them caught the crash).
- Do NOT speculate-and-patch. Each speculative patch requires Tornie
  to retest, costs CI minutes, and adds dead code.
- Do NOT bump version or push tag without Tornie's "go" message.
