# 1.0.277 — Need debugger access to identify crash site

## What I learned analyzing the .dmp files manually

I attempted to parse the Windows minidump files directly with Python
(struct-based parser). The format works in principle but the Microsoft
toolchain records the exception stream with a layout my parser did
not reproduce cleanly.

The crash report is consistent across versions 1.0.269 through 1.0.277
**with the same instruction** `cmp %dil, 0x1a(%rdx)` at offset `0x1400454F8`
through `0x1400455__` (the offset shifts as I add patches that move
adjacent code).

My hex-disassembly of offset `0x140045480` (the function containing the
crash) shows:
```
  140045494: movzbl 0x1a(%r9), %edi   ; read byte at offset 0x1a of arg 3
  1400454a1: mov (const_pool), %r8
  1400454ba: test %eax, %eax
  ...
  1400454eb: mov 0x9678(%r8,%rdx,8), %rdx   ; load entry from indexed table
  1400454f3: test %rdx, %rdx                 ; NULL check
  1400454f6: je   0x1400454fe                ; skip if null
  1400454f8: cmp  %dil, 0x1a(%rdx)          ; *** CRASH HERE ***
  1400454fc: je   0x140045513
  1400454fe: inc  %eax
  ...
  140045513: mov  $0x1, %al                ; function returns 1 if match
  ...
  14004550c: xor  %al, %al                 ; function returns 0 if no match
```

**This is a nested-loop index lookup**: scan a table indexed by
`%rdx*8 + 0x9678` (some fixed offset into a member array), read
houseID-typed field at offset 0x1a of each entry, compare against
the caller's houseID. The function returns true on first match, false
otherwise.

The nested loop bound `%rbp, %ebx, %r10d` (from offsets before 1400454F8)
matches `Map::isWithinBuildRange`'s `for (i, j) in BUILDRANGE loop`.
That diagnosis was correct. **But** the diagnostic SDL_Log I added in
1.0.277 did NOT fire in the crash. So either:

(a) the function being disassembled is a different one with the
same bytecode pattern (different class doing same HouseID lookup), OR
(b) my log code never executes because the crash occurs in a
different thread / before the map scan begins.

## What the patch in 1.0.277 achieved

1.0.277 added an `SDL_Log` inside `Map::isWithinBuildRange`'s nested
loop. Tornie's 1.0.277 log shows **no** `BUILDRANGE_DIAG:` lines before
the crash. So either isWithinBuildRange wasn't called OR the log line
was swallowed by the crash.

This either rules out `Map::isWithinBuildRange` as the crashing
function (case a), or points to a different timed crash (case b).

## What is needed to make progress

The .dmp files contain an exception stream but the function name is
stored in `.pdbs` (symbol files) that I do not have locally. **To
identify the exact crashing function** I need one of:

1. WinDbg on Windows. Open the .dmp, run `!analyze -v`. The output
   resolves addresses to function names using the embedded PDB
   signature and Microsoft's symbol server.
2. Visual Studio "Attach to Process" while running the game. The
   crash call stack will name the function.
3. A Windows debugger that maps dmp addresses to source via the
   project's `vc140.pdb` / `dunecity.pdb`.

Without these, I cannot identify the exact function. My bytecode
analysis narrows it to a "table-of-House-objects lookup with a TINY crash
instruction", but I cannot map that to a source-level function.
