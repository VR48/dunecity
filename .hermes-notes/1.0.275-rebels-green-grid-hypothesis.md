# Rebels green-grid bug — analysis

## Tornie's report

Tornie reports that the green-grid place-validity indicator appears for
Rebels on tiles where the actual `placeStructure` placement would fail.
The placement range check is inconsistent between the preview
(`Map::okayToPlaceStructure`) and the actual placement
(`House::placeStructure`).

## Code as of v1.0.275

### `Map::okayToPlaceStructure` (Map.cpp:363)

For each tile in the structure's bounding box, calls
`isWithinBuildRange(i, j, pHouse)`. If **at least one** tile is in range,
returns true. The green grid is drawn when this returns true.

### `Map::isWithinBuildRange` (Map.cpp:423)

For each tile in BUILDRANGE around (x, y), reads `tile->getOwner()`. If
the owner matches `pHouse->getHouseID()`, returns true. Else reads
`currentGame->getHouse(HOUSETYPE(ownerID))->getTeamID()` and compares
to `pHouse->getTeamID()` (the 1.0.254 fix).

### `House::placeStructure` (House.cpp:715)

The default case (regular structure, `itemID != Slab1/Road/Slab4`) only
checks:
1. Tile exists
2. No existing ground object on the tile (`hasAGroundObject`)
3. Otherwise sets location and creates the structure

**`House::placeStructure` does NOT call `isWithinBuildRange`**. There is
no teamID or build-range check at all. If `bForcePlacing == false`,
only the "existing object" check runs.

## Inconsistency

For Rebels (or any house with teamID != houseID), `okayToPlaceStructure`
and `placeStructure` diverge:
- `okayToPlaceStructure` may return true (at least one tile in BUILDRANGE
  is owned by a house on the same team).
- `placeStructure` ignores the BUILDRANGE entirely — only refuses if
  something is already there.

**Both functions should reach the same answer** for placement validity.

## Why "Invalid or occupied position" fires for Rebels

The 1.0.269 log shows:
```
SCENA001.INI:53: Invalid or occupied position for 'Soldier': '3985'!
House::placeStructure: houseID=7 itemID=2 xPos=14 yPos=9 ...
Briefing...
```

That's INIMap load, not player-initiated placement. `placeStructure` is
called from INIMapLoader (byScenario=true), and on `byScenario==true` the
function does NOT check anything special — it just constructs. The
"Invalid or occupied position" comes from `INIMapLoader` itself when
`placeStructure` returns nullptr.

But in player gameplay, the green grid is shown when
`okayToPlaceStructure` says "you can place here". For Rebels, the green
grid appears when `isWithinBuildRange` returns true (because at least
one tile in BUILDRANGE is owned by a friendly team). The actual
`placeStructure` allows placement if the tile is empty.

## So which is the bug?

Two possible interpretations:
- (A) The green grid is correct (Rebels can place), and the
      "placement fails" Tornie reports is actually a different bug
      (e.g. the placement is allowed but the game does not register it,
      a check elsewhere rejects it).
- (B) `placeStructure` should mirror `okayToPlaceStructure` and refuse
      placements where no tile is in BUILDRANGE. This would make the
      green grid match the placement result.

The user has implied (B) by saying "le bug green grid Rebels persiste".
A symmetric fix: call `isWithinBuildRange` for at least one tile in the
structure's footprint and return nullptr if all are out-of-range.

## Concrete fix (not yet applied)

In `House::placeStructure` default case (around House.cpp:820), add:

```cpp
if(bForcePlacing == false && pHouse != currentGame->getHouse(...)) {
    bool withinRange = false;
    for (int i = 0; i < newStructure->getStructureSizeX(); i++) {
        for (int j = 0; j < newStructure->getStructureSizeY(); j++) {
            if (currentGameMap->isWithinBuildRange(xPos+i, yPos+j, this)) {
                withinRange = true;
                break;
            }
        }
        if (withinRange) break;
    }
    if (!withinRange) {
        delete newObject;
        return nullptr;
    }
}
```

This makes `placeStructure` mirror `okayToPlaceStructure`'s reachability
logic. Place preview and actual placement become consistent.
