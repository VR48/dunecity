# Advanced Windtrap out-of-bound — analysis on Tornie's input

## Tornie's hypothesis

Tornie believes the vanilla Windtrap crash is caused by the **Advanced
Windtrap out-of-bound**. Reference: comment in
`src/FileClasses/GFXManager.cpp:1480`:

```
ObjPic_AdvancedWindTrap (DuneCity 1.0.254 single-static-PNG layout —
the 1.0.255 numImagesX=9 strip caused out-of-bounds blit because the
texture is a 10×7 frame sheet from
generateAdvancedWindtrapAnimationFrames, not 9 cells; reverted in
1.0.259 per 'out of bounding of Advanced Powerplant' user report)
```

## What I see in the code (after revert 1.0.259)

1. `AdvancedWindTrap::init` (AdvancedWindTrap.cpp:55-58):
   `numImagesX = 1; numImagesY = 1;`
   `firstAnimFrame = 0; lastAnimFrame = 0;`

2. `generateAdvancedWindtrapAnimationFrames(basePic)` (GFXManager.cpp:3547+):
   - Reads `fw = basePic->w`, `fh = basePic->h`
   - Creates a 32-bit RGBA `returnPic` of size
     `10 * fw × ceil((2 + 70) / 10) * fh = 10*fw × 8*fh`
   - Blits `basePic` into each cell of the 10×8 sheet
   - Writes light-pixel highlights per the hardcoded `lightPixels[]` table
   - Returns the 32-bit sheet

3. Caller path (GFXManager.cpp:3172-3175):
   ```
   rgbaSrc = SDL_ConvertSurfaceFormat(src, SDL_PIXELFORMAT_RGBA32, 0);
   src = rgbaSrc.get();
   objPicTex[id][house][z] = convertSurfaceToTexture(generateAdvancedWindtrapAnimationFrames(src).get());
   ```
   `src` is always 32-bit RGBA before being passed in.

## My read

- The Advanced Windtrap layout is internally consistent. The structure
  reads `numImagesX=1`, the sheet has 70 frames, and the structure
  accesses `indexX = 0, indexY = 0` → cell 0 → no out-of-bounds.
- **If** `basePic` has `w == 0` or `h == 0`, then `sizeX = 0` or
  `sizeY = 0`, `SDL_CreateRGBSurface` returns NULL, `generateAdvancedWindtrapAnimationFrames`
  returns `{}`, and `convertSurfaceToTexture(NULL)` would crash.
- **If** `basePic` has `w == 0` or `h == 0` because the loader failed to
  find `Tornie_AdvancedWindtrap_gfx.png` and the fallback
  `super_power_plant.png` is also missing, then `superSrc` is empty,
  `atlas` (createRGBSurface 0×0) is empty, no SDL_BlitScaled fires, but
  the **house-copy loop** still runs `SDL_ConvertSurface` on the empty
  HARKONNEN surface — this might trigger a NULL deref in SDL
  internals.

## What I think Tornie's intuition is signalling

**Tornie is probably right that the Advanced Windtrap is the site, but
not necessarily the layout problem.** The OOB happens because the
fallback sprite load fails silently (no `Tornie_AdvancedWindtrap_gfx.png`
or `super_power_plant.png` available in the vanilla PAK the user is
running), and the loader's house-copy loop dereferences a NULL surface.
