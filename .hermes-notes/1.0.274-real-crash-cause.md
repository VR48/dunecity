# Vanilla crash — REAL root cause identified (do NOT patch until validated)

## Source of truth

Tornie sent the **1.0.274 vanilla crash log** on 2026-07-04 09:47. The
log contains a load-bearing diagnostic line that no previous log had:

```
Game::init: scenario load failed (gameType=3, mission=22):
D:\a\dunecity\dunecity\src\misc\draw_util.cpp:608:
mapSurfaceColorRange(): Cannot copy image
(SDL_ConvertSurface failed: Empty destination palette)!
```

## What this tells us

1. The crash site is **`src/misc/draw_util.cpp:608` `mapSurfaceColorRange()`**.
2. `SDL_ConvertSurface` fails with `Empty destination palette` — the
   destination surface was created with a 0-entry palette.
3. This is the actual root cause of the `0xC0000005 at 0x54F8` crash
   in `dunecity.exe`. `SDL_ConvertSurface` throws a `std::runtime_error`
   on failure, which (depending on the build) becomes `0xE06D7363` (a
   C++ exception), and the exception handler unwinds through the same
   chain of frames that produced the `0x54F8` offset.
4. `mission=22` is the last mission in **any** campaign (Harkonnen,
   Atreides, Ordos, Fremen, Sardaukar, Mercenary, Rebels).

## Why the cosmetic patches did not help

The 1.0.266 → 1.0.273 source-rect clamps in `blitToScreen`,
`drawSmoke`, `InfantryBase::blitToScreen` were all **downstream** of
`mapSurfaceColorRange()` — they guard against OOB reads in *successful*
`SDL_RenderCopy` calls, but `mapSurfaceColorRange` throws *before*
`SDL_RenderCopy` is reached. So the clamps never fire.

## Why the render offset is constant `0x54F8`

The `0x54F8` offset likely corresponds to `mapSurfaceColorRange`'s
clean-up path: after `SDL_ConvertSurface` fails, the function
dereferences a NULL `dst->format->palette` (because the palette is
empty by definition of the failure). The `cmp %dil, 0x1a(%rdx)` is
loading `dst->format->palette->ncolors` (the field at offset 0x1a in
an `SDL_Palette` struct) on a NULL palette pointer.

## Mission 22 specifically

Harkonnen mission 22 = `DUNE1` (campaign 22) which was the last
   mission loadable when Tornie plays vanilla mod.

   The user-mentioned `Now playing DUNE9.ADL!` and `DUNE18.ADL!`
   in *earlier* crash logs suggests they load mission 22 (or whatever
   mission 22 maps to per mod) and the map surface fails because the
   Tornie mod's `mods/Tornie/data/` does not ship the custom palette
   that `mapSurfaceColorRange` needs.
